# gemm_sweep — VersaCore GEMM RTL cycle-count characterization

This workload runs a **whole sweep of (M, K, N, array_shape) configs in a
single DFG** and measures per-config cycle counts via the
`BINGO_TRACE_GEMM_FULL_RUN_*` and `BINGO_TRACE_GEMM_MIN_RUN_*` markers
embedded in the device kernels — no host-side mcycle bracketing.

## Per-config DFG

For every config in `CONFIGS` (inside `main_bingo.py`) the DFG emits:

```
Load_A → Load_B → GemmFull → GemmMin → Check_D
```

- **GemmFull** (`__snax_bingo_kernel_gemm_full`) — configures the VersaCore
  streamers for `(M, K, N, array_shape)` and runs the GEMM.  Its cycle count
  includes the reconfiguration overhead.
- **GemmMin** (`__snax_bingo_kernel_gemm_minimal`) — reuses the streamers
  that GemmFull just set up (only reprograms the A/B/C/D base addresses) and
  reruns with the same data.  This is the steady-state cost you'd see for a
  sequence of identically-shaped GEMMs.

The host `Check_D` kernel verifies the top 256 B of the result against the
per-config golden generated by `gemm_datagen.py`.

## Adding new configs

Edit the `CONFIGS` list at the top of `main_bingo.py`.  Each entry is
`(M, K, N, array_shape)` in mesh-tile units.  The workload will:

- Pre-compute the per-config A/B/D dimensions from the hjson hw config
- Check that A+B+D fits under `L1_SIZE_KB` (default 500 KB) for every config
- Size the shared L1 buffers to the max across configs
- Emit per-config arrays in `gemm_data.h`

## Files

| File | Purpose |
|------|---------|
| `main_bingo.py`     | CONFIGS list + DFG builder + configs.json dump |
| `gemm_datagen.py`   | Per-config A/B/D golden-data generator         |
| `Makefile`          | Build rules (no CLI overrides needed)          |
| `configs.json`      | Generated: list of configs for the shell driver |
| `gemm_data.h`       | Generated: per-config data header              |
